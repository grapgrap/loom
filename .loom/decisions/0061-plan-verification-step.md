# plan에 사용자 동반 검증 단계를 추가한다

## 맥락

plan은 초안을 에이전트가 작성한 뒤 사용자의 승인만으로 확정되는 일괄 처리 구조였다. 분해와 의존관계 식별을 에이전트가 독자적으로 수행하고 "이 정도면 실행 가능하다"는 종결 판단까지 내린 뒤 산출물을 한 번에 내놓았다. 그 결과 두 가지 문제가 드러났다.

첫째, 초안이 실제로 실행 가능한 수준에 도달했는지를 보완하는 장치가 없었다. 둘째, 완성된 plan을 그대로 넘기지 않고 각 태스크를 사용자와 한 번 더 짚으니 plan의 품질이 눈에 띄게 좋아졌다.

task의 방향성 합의가 이 보완을 일부 수행하기는 한다. 그러나 task의 합의는 작업 직전에, 단일 태스크를, 코드 수준까지 내려가 확인하므로 실현 가능성 점검에는 탁월하나 큰 그림을 잃고 시점이 늦다. 한 태스크에서 "실행 불가"가 드러나도 그것이 전체 plan에 어떻게 번지는지 보기 어렵다.

## 결정

plan 스킬에 "검증" 단계를 추가한다. 초안이 산출된 직후, 확정 전에 사용자와 함께 각 태스크의 방향성을 확인하는 단계다.

- 주체는 사용자와 에이전트의 공동 작업이다. 에이전트 단독 점검이 아니다.
- 대상은 전 태스크이며, 큰 그림(전체 순서와 의존관계)을 유지한 채 본다.
- 깊이는 각 태스크가 닿을 거점이 실재하고 접근 가능한지, 의존이 끊기지 않는지를 확인하는 수준에 머문다. 코드 수준의 구체적 변경 방식은 task에 남긴다.
- 더 깊은 해상도가 필요한 태스크는 검증에서 파고들지 않고, shape 라우팅이나 task 위임 표식만 남긴다.
- 검증은 마지막 통과 의례가 아니라 교정 루프다. 거점 부재, 의존 단절, 변경의 부당함이 드러나면 분해나 의존관계 식별로 되돌아가 조정하고 재검증한다.

기존 plan 스킬의 "해상도 점검"(에이전트 단독, 분해 도중, 단발)을 이 검증 단계로 흡수하여 승격한다.

### 전제 조건

- 사용자가 변경 범위를 정확히 인지하지 못하면 그 변경이 옳은지 판단할 수 없다: 검증을 사용자 동반으로 둔다.
- 초안을 에이전트가 작성하는 것 자체는 문제가 아니다: 검증은 초안을 대체하지 않고 보완한다.
- 한 태스크의 실행 가능성 문제는 전체 plan에 파급된다: 검증을 단일 태스크가 아니라 큰 그림에서 수행한다.

### 제약 조건

- 초기 plan이 거친 것은 자연스러우며 모든 태스크에 높은 해상도를 강제하면 plan 작성 비용이 불필요하게 커진다 (decision 0013): 검증의 깊이를 거점 확인 수준으로 제한하여 양립시킨다.
  - 제약이 없었다면: 검증에서 각 태스크를 코드 수준까지 깊이 합의했을 것이다.
- task가 작업 직전의 구현 수준 합의를 담당한다 (decision 0024): plan의 검증은 거시(거점·작성 직후·큰 그림), task의 방향성 합의는 미시(코드·작업 직전·단일 태스크)로 해상도를 분담한다.
  - 제약이 없었다면: plan 검증과 task 합의가 같은 깊이에서 중복되었을 것이다.

## 대안

### 에이전트 자기 점검으로 검증

- 설명: 검증을 에이전트가 단독으로 수행하고 결과만 사용자에게 보고한다.
- 트레이드오프: 사용자가 plan으로 변경되는 범위를 정확히 인지하지 못해, 그 변경이 옳은지 판단할 근거를 갖지 못한다.

### 검증을 task의 방향성 합의에 일임

- 설명: plan에는 검증 단계를 두지 않고, 실행 가능성 확인을 task의 방향성 합의에 맡긴다.
- 트레이드오프: task의 합의는 단일 태스크·작업 직전·코드 수준이라 큰 그림에서 실행 가능성을 보지 못하고, 문제 발견 시점이 늦다.

### plan 작성 시점에 구체도를 강제

- 설명: spec-kit 방식으로 모든 태스크가 높은 해상도의 중간 산출물을 갖도록 강제한다.
- 트레이드오프: decision 0013이 이미 기각한 방향이다. 초기 plan이 거친 것은 자연스러우며, 일괄 강제는 비용을 키운다.

### 검증 대상을 위험·불확실한 태스크로 선별

- 설명: 전 태스크가 아니라 위험하거나 불확실한 태스크만 검증한다.
- 트레이드오프: 선별에서 빠진 단계의 문제를 놓치며, 사용자가 plan 전체의 변경 범위를 인지하게 한다는 목적을 달성하지 못한다.

## 영향

- [[plan]] - 책임에 각 태스크의 사용자 동반 검증이 추가되고, 스킬에 검증 단계가 생기며 기존 해상도 점검이 흡수된다.
