# flow를 조건부 축으로 재조정하고 concept을 조정자·계산을 포함하는 노드로 명확히 한다

## 맥락

decision 0048은 flow를 concept과 "대등한 축"으로 도입했다. 그러나 새 프로젝트(expression)에 적용한 결과, flow가 "만들어야 한다는 압박"에 못 이겨 빈약하게 산출되는 현상이 관찰되었다. 편향 없는 서브에이전트 검증에서도, 평가·검증처럼 여러 입력을 받아 동작하는 것을 일관되게 처리기 concept으로 세우고, flow는 그 동작을 재기술한 빈 껍데기로 만들었다.

DDD를 재검토한 결과 세 가지가 드러났다:

- aggregate(일관성 경계, 불변식 보호)는 flow가 아니라 concept에 대응한다. concept은 요청을 받아 규칙을 지키며 상태를 전이시키고 사건을 내는 노드다.
- flow가 대응하는 것은 domain message flow / saga이며, 협력이 어느 단일 노드에도 귀속되지 않고 흩어져 빚어질 때만 환원 불가능한 일급이 된다. 협력을 떠맡는 조정자가 있으면 그것은 concept이다.
- domain message flow는 영속 산출물이 아니라 경계를 검증하는 활동(렌즈)이다. 흐름을 그리는 것과 flow를 남기는 것은 다르다.

## 결정

flow의 위상을 "concept과 대등한 축"에서 "concept이 기본 분해 축이고, flow는 협력이 흩어질 때 출현하는 조건부 축"으로 재조정한다.

- **concept**은 자기 상태와 규칙을 지니고 요청에 따라 상태를 전이시키는 노드다. 협력을 관장하는 조정자도, 여러 입력을 받아 한 결과로 닫히는 계산도, 단일 노드로 이름 붙는 한 concept이다.
- **flow**는 협력을 떠맡는 단일 노드에 이름을 붙일 수 없을 때 — 협력이 어디에도 귀속되지 않고 흩어질 때 — 만 남긴다.
- flow의 단계 사이를 흐르는 것은 상태를 전이시키는 사건이며, 데이터 변환의 연쇄(계산)는 flow가 아니다.
- shape에서 흐름을 도출하는 일은 경계를 비추는 렌즈(활동)이고, flow 문서 산출은 판별을 통과할 때만 일어난다.

판별 기준: "이 협력을 떠맡는 단일 노드에 이름을 붙일 수 있는가." 붙으면 concept, 못 붙으면 flow.

### 전제 조건

- 협력의 조정 주체가 단일 노드에 귀속되는지 여부가 설계 시점에 판별 가능하다

### 제약 조건

- flow가 조건부가 되어 산출 빈도가 낮아진다 — 0048이 해소하려던 "흐름의 표류"가 일부 도메인에서는 flow 없이 concept으로 흡수된다
  - 제약이 없었다면: 모든 도메인 간 흐름을 flow로 남겼을 것이나, 그러면 expression처럼 빈 flow가 강제된다

## 대안

### 0048의 "대등한 축"을 유지

- 설명: flow를 concept과 대등하게 항상 산출한다
- 트레이드오프: 협력이 없거나 조정자에 귀속되는 도메인에서도 빈 flow가 강제된다 (expression에서 입증된 압박)

### 조정자가 있어도 흐름을 렌즈로 남긴다 (넓은 쪽)

- 설명: 단일 concept으로 한눈에 보이지 않는 흐름은 이해를 돕기 위해 flow로 남긴다
- 트레이드오프: 편의를 주지만 "flow를 만들어야 한다는 압박"의 문을 다시 연다. 많은 flow의 조망은 index와 aeira 그래프의 몫으로 미룬다
