# 각 스킬은 rule을 자기 산출물 관점에서 항목별로 활용한다

## 맥락

calibrate에서 누적 감사를 진행하면서 빈 공간이 드러났다: shape, plan, task, calibrate는 Step 0에서 rule을 로드하지만, 본문 어디에서도 rule을 활용하는 절차가 명시되지 않았다. review만 명시적 검증 단계를 가진다.

rule concept이 "plan과 task가 Step 0에서 자동 로드한다"고 명시하므로, 로드만 하고 본문에서 활용하지 않는 것은 모순이다.

0045에서 rule이 "프로젝트 종속의 비기능 요구사항"으로 좁혀지면서, 활용 절차도 그에 맞는 형태로 정의할 수 있다. 비기능 요구사항은 작동 시점이 다양하다 — 설계 시점부터 배경에 깔리는 항목도 있고, 코드 작성 시점에만 작동하는 항목도 있다. 따라서 각 스킬이 자기 산출물 관점에서 rule의 작동 여부를 판단하는 형태가 자연스럽다.

## 결정

각 스킬은 로드한 rule 각각에 대해 자기 산출물 관점에서 작동 여부를 판단한다. rule에 작동 시점 메타데이터를 부여하지 않는다 — 작동 시점은 rule 정의의 추상화 수준에서 자연스럽게 드러난다.

각 스킬의 활용 형태:

- **shape**: rule이 새 concept에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 설계한다. 충돌이 발견되면 사용자에게 보고한다 (설계 변경 또는 rule 변경 결정).
- **plan**: rule이 태스크에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 태스크를 분해한다.
- **task**: rule이 코드에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 코드를 작성한다.
- **review**: 구현이 적용되는 rule을 만족하는지 사후 검증한다 (이미 정의됨, 변경 없음).
- **calibrate**: rule 자체를 누적 점검하고 신규 항목을 발견한다 (이미 정의됨, 변경 없음).

작동 시점이 다양한 rule은 자연스럽게 여러 스킬에서 활성화된다.

### 전제 조건

- rule 정의의 추상화 수준이 작동 시점을 자연스럽게 드러낸다: 정의의 결이 다르면 활용 시점에 판단 가능
- 각 스킬이 자기 산출물의 형태를 알고 있다: shape는 concept을, plan은 태스크 분해를, task는 코드를 산출한다 — 그 산출물의 형태에 rule이 영향을 미치는지 판단 가능

## 대안

### rule에 작동 시점 메타데이터 부여

- 설명: 각 rule 항목에 "설계 시점" 또는 "구현 시점" 메타데이터를 두고, 스킬은 해당 시점의 rule만 로드
- 트레이드오프: 자동화는 강해지지만 메타데이터 관리 비용 추가. 가장자리 케이스(설계·구현 모두 영향)에서 분류 모호. rule을 단일 엔티티로 유지하기로 한 결정과도 약하게 충돌

### shape/plan에서 rule 활용 없음 (review/task만)

- 설명: rule은 코드 작성 시점에만 적용되는 강제로 한정하고, shape/plan은 rule을 무시
- 트레이드오프: rule concept이 plan/task의 자동 참조를 명시한 것과 모순. 비기능 요구사항 중 설계 시점에 작동하는 항목이 review에서 사후 발견되어 shape 재진입 비용 발생

## 영향

- [[rule]] - 활용 절차가 동작 원리에 추가된다
- shape - Step에 rule 활용 절차 추가
- plan - Step에 rule 활용 절차 추가
- task - Step에 rule 활용 절차 추가
