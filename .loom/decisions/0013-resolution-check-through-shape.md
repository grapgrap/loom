# 해상도 점검과 해소는 shape를 통해 수행한다

## 맥락

plan 태스크를 실행할 때, 에이전트가 방향성 수준의 판단(인터페이스 형태, 데이터 흐름 설계, 추상화 전략)을 자의적으로 내리는 문제가 발생했다.
expression-library의 dispatch table 태스크에서 데이터 구조, 조회 키 형태, 타입 안전성 전략을 에이전트가 재량으로 결정한 것이 구체적 사례이다.

초기 plan이 거친 해상도를 갖는 것은 자연스러운 일이다. 문제는 거친 plan을 그대로 실행에 넘기는 것이다.
인간 팀에서는 실행 전에 논의를 통해 해상도를 올리는 과정이 자연스럽게 끼어드는데, 에이전트에게는 이 과정이 부재했다.

## 결정

해상도가 부족한 태스크를 발견하면 shape를 통해 해소한다. shape는 기능 단위뿐 아니라 태스크 단위에서도 작동한다.

- 에이전트가 태스크 실행 전에 "방향성 수준의 판단을 해야 하는가"를 인식하면, 자의적으로 진행하지 않고 사용자에게 알린다.
- 사용자가 해상도 부족을 인식하면 shape를 호출한다.
- shape의 triage가 입력의 스케일을 판단하므로, 작은 발견이면 빠르게 통과하고 큰 동굴이 열리면 전체 프로세스를 수행한다.

### 제약 조건

- shape의 기존 메커니즘(triage, 도메인 탐색, 설계 방향 수립)이 이미 해상도 해소에 적합하다: 새로운 커맨드가 불필요하다

### 전제 조건

- 태스크 수준의 발견이 concept 변경으로 확대될 수 있다: shape가 이를 자연스럽게 다룬다
- 발견의 스케일은 사전에 예측할 수 없다: triage가 사후적으로 조절한다

## 대안

### 새로운 implement 커맨드 도입

- 설명: 실행 코디네이션을 전담하는 별도 커맨드를 만든다
- 트레이드오프: shape의 기존 메커니즘과 중복되며, shape가 이미 해상도 해소에 적합하다

### Plan 작성 시점에 구체도를 강제

- 설명: spec-kit 방식으로 plan이 contracts, data-model 등 중간 산출물을 추가로 생성하도록 한다
- 트레이드오프: 초기 plan이 거친 것은 자연스러운 일이며, 모든 태스크에 높은 해상도를 강제하면 plan 작성의 비용이 불필요하게 증가한다

## 영향

- [[plan]] — 태스크가 작성 시점의 해상도를 반영하며, 실행 전 shape를 통한 해소가 필요할 수 있음을 인정한다
- [[shape]] — 태스크 단위 입력을 수용하도록 triage와 라우팅을 확장한다
