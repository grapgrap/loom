# rule을 프로젝트 종속의 비기능 요구사항으로 정의를 좁힌다

## 맥락

0029에서 셋(원칙·룰·컨벤션) 분화가 이루어졌고, 0038에서 둘(rule·convention)로 강제성 축으로 재정의되었다. 0037에서 project scope의 principle 엔티티가 폐기되었다.

calibrate에서 누적 감사를 진행하면서 두 가지 모호함이 드러났다:

1. **rule 안의 깊이 다양성**: rule이 "강제성 무게"만으로 정의되면 행동 강제와 설계 판단이 같은 엔티티에 섞인다. 후자는 원칙에 가깝다는 직관이 있었다.
2. **원칙의 보관 자리**: 원칙이 설계에 도움을 주지만, 프로젝트에 영속적으로 보관될 정도의 무게를 가지는가? 0037에서 폐기된 자리를 다시 부활시킬 정당성이 부족하다 — decision 누적과 스킬 본문 녹임(0027)이 그 역할을 이미 흡수하고 있다.

이 두 점검을 거치면, rule의 정체성이 모호한 원인은 정의가 너무 광의이기 때문이며, 정의를 좁히는 방향이 자연스럽다. 원칙은 rule이 흡수하지 않으며, 사용자 영역에 머문다는 0037의 결정이 유지된다.

## 결정

rule을 "프로젝트 종속의 비기능 요구사항"으로 정의를 좁힌다.

- rule은 이 프로젝트에서만 유의미한 비기능 요구사항을 명시한다
- 프로젝트 횡단의 설계 판단(원칙)은 rule이 아니다. 원칙은 사용자 본인의 영역에 머문다 (0037 유지)
- rule 안에 깊이가 다양한 항목이 공존할 수 있다. 별도 하위 분류나 작동 시점 메타데이터를 두지 않는다

원칙이 프로젝트에서 활성화되는 경로는 다음 두 가지로 충분하다:
- 스킬 본문에 녹은 형태 (0027)
- decision 누적이 보존하는 설계 판단의 맥락

### 전제 조건

- decision 누적과 스킬 본문 녹임이 원칙의 작동을 충분히 보장한다: 0037의 전제와 동일
- "비기능 요구사항"이라는 범주가 아키텍처 제약 같은 항목을 포섭한다: 기능을 어떻게 구현할 것인가에 대한 제약은 모두 비기능 요구사항으로 분류 가능

## 대안

### rule을 작동 시점으로 분리

- 설명: rule을 "설계 rule"과 "구현 rule"로 하위 분류
- 트레이드오프: 0029의 셋 분화를 0038에서 둘로 축소한 경험과 역행한다. 분화 비용이 크고, 0037이 폐기한 자리(깊은 판단의 별도 엔티티)를 다시 만드는 셈

### rule 정의를 광의로 유지

- 설명: "강제적 합의 전체"로 두고 정밀화하지 않음
- 트레이드오프: rule 안의 깊이 다양성과 원칙과의 경계 모호함이 해소되지 않는다. calibrate에서 동일한 모호함이 다시 발견될 가능성

### project scope에 원칙 엔티티 부활

- 설명: 0037을 부분 뒤집어 rule과 원칙을 project scope에 공존시킴
- 트레이드오프: 원칙이 프로젝트에 영속 보관될 정당성이 약하다. decision 누적과 스킬 본문 녹임이 이미 그 역할을 한다

## 영향

- [[rule]] - 정의가 "프로젝트 종속의 비기능 요구사항"으로 좁아진다. 강제성 무게(0038) 위에 본질이 추가된다
- [[convention]] - 비교 축이 분명해진다 (rule: 강제, convention: 권장 — 둘 다 비기능 요구사항을 다룰 수 있음)
- [[decision]] - decision 누적이 원칙의 작동 경로 중 하나임을 명시 (0037과 일관)
