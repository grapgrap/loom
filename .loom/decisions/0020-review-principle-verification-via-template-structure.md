# Review에서 원칙의 적용/테스트/경계 구조를 활용한 검증

## 맥락

review 스킬의 Step 1에 "원칙 위반: 현재 원칙에 어긋나는 부분이 있는가"라는 지시가 있지만, 실제로 에이전트가 원칙을 근거로 리뷰한 적이 한 번도 없었다.
concept 검증은 잘 작동하는 반면 원칙 검증은 작동하지 않는 이유를 분석한 결과, concept은 대조 방법이 구체적인 반면 원칙은 "위반이 있는가?"라는 추상적 질문만 있어 에이전트가 실행하지 못하는 것이 원인이었다.

## 결정

review 스킬의 원칙 검증을 보강한다. 원칙 자체를 변경하지 않고, review 스킬이 원칙의 기존 구조(적용/테스트/경계)를 활용하도록 검증 절차를 명시한다.

### 제약 조건

- 원칙은 "판단의 근거가 되는 믿음"이라는 정체성을 유지해야 한다: 원칙에 검증 방법을 넣으면 규칙으로 변질된다
- concept에 구현 세부사항을 넣지 않는 것처럼, 원칙에도 검증 절차를 넣지 않는다: 소비하는 쪽(review 스킬)이 방법을 알아야 한다

### 전제 조건

- 원칙 템플릿이 이미 적용/테스트/경계 섹션을 포함하고 있다: 에이전트가 활용할 구조가 이미 존재한다
- review 스킬이 Step 0에서 원칙을 로드하고 있다: 데이터 접근은 이미 해결되어 있다

## 대안

### A. 원칙 concept 자체에 "위반 시그널"을 추가

- 설명: 각 원칙 파일에 "코드에서의 위반 시그널" 섹션을 추가하여 원칙이 스스로 검증 방법을 포함
- 트레이드오프: 원칙이 규칙으로 변질될 위험. concept에 구현 세부사항을 넣지 않는 것과 같은 이유로 기각

## 영향

- [[principle]] - 원칙 자체는 변경 없음. 기존 적용/테스트/경계 구조가 review에서 소비됨
