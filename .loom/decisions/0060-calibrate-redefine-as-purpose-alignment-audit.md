# calibrate를 목적 정렬 감사로 재정의한다

## 맥락

calibrate는 "concept과 코드의 누적 괴리를 감사"하는 스킬이었다(0011). 이후 직접 수정을 금지하고 shape 재진입으로 위임하는 report-only 성격이 확정되었고(0032), rule 변경 책임도 제거되어 누적 감사·리포트 전용으로 한정되었다(0054).

현행 calibrate의 감사는 정합성 기반이다 — concept과 코드, concept과 decision을 짝지어 비교하고 어긋난 지점을 짚는다. 그러나 loom은 설계 지식 체계이며, 설계 지식 체계에서 더 값어치 있는 감사는 "지도와 영토가 어긋났는가"가 아니라 "지도가 목적지로부터 멀어지는 방향으로 자라났는가"이다. 프로젝트가 concept을 쌓아가면서, 개별 concept의 목적이 프로젝트 전체의 목표와 더는 이어지지 않는 채로 누적될 수 있다. 정합성 감사는 이를 잡지 못한다 — 서로 어긋나지 않으면서도 목표와 무관하게 자란 concept은 정합적이기 때문이다.

## 결정

calibrate를 "프로젝트가 쌓아 온 목표에 현재 concept들의 목적이 정합하는지를 목적의 위계로 가늠하는 문서 감사"로 재정의한다.

**준거**: index.md의 프로젝트 목표 진술. index concept이 이미 "프로젝트의 정의, 범위, 핵심 기능을 지도의 범례로 포함한다"고 허용하는 역할에 프로젝트 목표를 필수 항목으로 승격한다. calibrate는 이 목표를 감사의 닻으로 삼는다.

**목표 부재 시 동작**: calibrate는 감사에 앞서 index.md에서 프로젝트 목표를 읽는다. 목표가 부재하면 concept 간 목적의 정합성(중복, 과적, 고립)을 점검하되, 목표 대비 정렬은 건너뛴다. 목표 부재 자체는 리포트에서 "목적 위계의 꼭대기 부재"로 별도 알린다.

**코드 제거**: 코드 대비 감사를 calibrate의 범위에서 제거한다. calibrate는 문서 계층(index, concepts, flows, decisions) 내부의 감사로 한정한다. 세션을 건너 쌓인 코드 괴리는 세션별 review와 개발자에게 맡긴다.

**감사 차원 재편**:

1. **목적 연결의 무결성** (신설) — 각 concept의 목적이 프로젝트 목표로 이어지는가. 목표와 연결되지 않는 concept, 또는 목적이 선언되었으나 프로젝트 목표에서 대응물이 없는 concept을 식별한다.
2. **전제·제약 무효화** (존속, 관점 전환) — 전제가 무효화된 concept은 목표에 복무하던 근거가 무너진 것이다. "전제가 유효한가"에서 "이 concept이 목표에 복무하는 근거가 아직 서 있는가"로 관점을 전환한다.
3. **분화의 옵션 가치 재평가** (존속) — 쓰임을 얻지 못한 분화는 목표에 복무하지 않는 누적이다.
4. **concept과 decision의 괴리** (존속, 관점 전환) — decision이 방향을 전환했는데 concept 목적이 갱신되지 않은 경우, 목적 위계에 단절이 생긴 것이다.

기존 "경계 이탈" 차원 중 코드 대비 항목은 코드 제거에 따라 삭제한다. 문서 간 경계 이탈(concept 간 책임 중첩·모호)은 (1)에 흡수한다 — 두 concept이 같은 목적을 주장하거나 한 concept의 목적이 과적된 것은 목적 연결 실패의 한 양상이다.

**불변**: report-only 성격(0032/0054)은 유지한다. calibrate는 감사 결과를 리포트할 뿐 직접 수정하지 않으며, 교정 방향은 사용자가 shape 재진입으로 판단한다.

### 전제 조건

- index.md가 프로젝트 목표를 담는 역할을 갖는다: index concept의 범례 역할에 프로젝트 목표를 필수로 승격함으로써 충족한다
- 목적 위계가 감사의 유효한 준거 틀이다: 설계 지식 체계에서 각 노드의 존재 이유는 상위 목표에 복무하는 것이며, 이 복무 관계의 무결성을 점검하는 것이 정합성 점검보다 근본적이다

### 제약 조건

- 코드 대비 감사를 제거하므로, 세션을 건너 쌓인 설계·구현 괴리의 전담 감사가 없어진다
  - 제약이 없었다면: calibrate가 문서와 코드를 모두 감사했겠으나, loom은 설계 지식 체계이므로 calibrate가 설계 계층 내부의 일관성에 집중하고 코드 충실도는 세션별 review와 개발자에게 맡기는 분담이 경계를 더 분명히 한다

## 대안

### 준거를 index.md에 고정하지 않고 에이전트가 귀납

- 설명: calibrate가 전체 문서를 읽고 "이 프로젝트가 무엇을 향해 가고 있는가"를 종합한 뒤, 그 종합에 비추어 감사한다
- 트레이드오프: 에이전트의 귀납에 해석 편향이 개입하면 감사의 닻이 실행마다 흔들린다. 명시적 목표 진술이 있어야 감사가 재현 가능하다

### 정합성 감사를 유지하면서 목적 정렬 차원만 추가

- 설명: 기존 네 차원(코드 대비 포함)을 유지하고 목적 연결의 무결성을 다섯 번째 차원으로 추가한다
- 트레이드오프: 코드 대비 감사와 목적 정렬 감사는 준거 틀이 다르다(쌍별 비교 vs 위계 순회). 한 스킬에 두 준거 틀을 섞으면 감사의 초점이 분산되고, calibrate의 변경 이유가 둘 이상이 된다
