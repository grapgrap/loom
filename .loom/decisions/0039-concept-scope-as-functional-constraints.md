# concept이 다루는 범위를 기능 제약으로 한정한다

## 맥락

새 구조(rule + decision 누적이 principle 역할 흡수)에서 비기능 제약사항이 어떻게 표현되는지를 점검하던 중, "비기능 제약을 concept으로 표현 가능"이라는 처리가 후보로 올라왔다.

사용자가 교정: concept은 본질적으로 기능 제약(도메인 엔티티)을 다루는 자리다. 비기능 제약은 concept의 정의에 전제 조건으로 부수적으로 등장할 수는 있지만, concept으로 정의하는 것은 권장되지 않는다.

이 교정은 concept의 정체성을 더 명확히 한다.

## 결정

concept이 다루는 범위를 기능 제약으로 한정한다. 비기능 제약사항은 다음 두 자리로 분류한다:

- **추상적 방향성 (trade-off 우선순위)** → decision (누적)
  - 예: "이 프로젝트는 일관성보다 가용성을 우선한다"
- **구체적 강제 사항** → rule
  - 예: "API endpoint의 응답시간 ≤ 100ms", "비밀번호는 bcrypt로 해시"

concept의 정의에 비기능 제약이 전제 조건으로 부수적으로 등장하는 것은 허용된다. 그러나 비기능 제약 자체를 concept으로 정의하는 것은 권장되지 않는다.

### 전제 조건

- 비기능 제약의 두 결(추상적 방향성과 구체적 강제)이 분리 가능하다: 같은 제약이 추상적 형태로 표현될 때와 구체적 형태로 표현될 때 자리가 다르다
- decision의 누적이 추상적 방향성을 보존하기에 충분하다: 0037의 전제와 동일

## 대안

### 비기능 제약을 concept으로 표현 가능하게 허용

- 설명: concept의 다루는 범위를 도메인 엔티티 + 비기능 제약으로 확장
- 트레이드오프: concept의 정체성이 모호해진다. concept은 "프로젝트가 다루는 세계의 구성 요소"인데, 비기능 제약은 그 세계의 구성 요소가 아니라 그 세계가 작동하는 환경 조건이다

### 비기능 제약을 위한 별도 엔티티 도입

- 설명: constraint 같은 새 엔티티를 두어 비기능 제약을 다룬다
- 트레이드오프: rule + decision 두 자리로 분류 가능한데 별도 엔티티는 중복이다. 새 엔티티의 진입·자동 참조·관리 비용이 추가된다

## 영향

- [[concept]] - 다루는 범위 명시 (기능 제약), 비기능 제약은 전제 조건으로만 부수적 등장
- [[rule]] - 비기능 제약 중 구체적 강제 사항이 rule의 영역으로 자리잡음
- [[decision]] - 비기능 제약 중 추상적 방향성이 decision의 영역으로 자리잡음
