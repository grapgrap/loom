# rule과 convention의 정체성을 강제성 축의 양극으로 재정의한다

## 맥락

0037에서 rule은 "원칙의 영역적 발현, 조건과 행동의 매핑"으로, convention은 "합의된 표현의 통일"로 정의되었다. 이 분화 축은 "표현·구조의 종류"였다.

코드 작성 위임의 합의 잔고를 분류하면서 0037 정의가 두 종류의 모호성을 드러냈다:

1. **rule과 convention의 경계 모호**: 사용자가 코드 작성 위임에서 부족함을 느끼는 항목들(모듈 구성 방법, 인터페이스 명명규칙, 함수 시그니처 구성, 선호 라이브러리)을 분류해 보면, 다수가 "원칙의 영역적 발현"이라기보다 단순히 합의된 통일이다. rule과 convention 어느 쪽에 속하는지 모호하다
2. **convention 정의의 영역 부족**: "합의된 **표현**의 통일"은 명명규칙은 흡수하지만 모듈 구조나 라이브러리 선택을 흡수하지 못한다. 선호 라이브러리는 표현이 아니라 도구 선택이다

사용자의 인식 — "어떤 룰들은 항상 지켜져야 하는 것이고 어떤 것들은 선호에 가까운 것들이 있다" — 은 분화의 새 축을 제시한다: 강제성의 무게.

## 결정

rule과 convention의 정체성을 강제성 축의 양극으로 재정의한다. 0037의 셋 분화는 둘로 축소되며, principle 엔티티 폐기는 0041에서 별도로 처리된다.

- **rule**: 항상 지켜져야 하는 강제적 합의 (강제성 축의 무거운 자리)
- **convention**: 강하게 권장되지만 대체 가능한 형태 (강제성 축의 가벼운 자리)

convention의 영역을 표현 너머로 확장한다 — 표현·구조·도구의 합의된 통일을 모두 포함한다. 선호 라이브러리, 모듈 구성 패턴 등이 자연스럽게 들어간다.

조건(영역적 또는 상황적)은 rule이나 convention의 정의 안에 포함한다 — 별도 엔티티로 분리하지 않는다. "service 레이어에서는", "비밀번호를 다룰 때는" 같은 조건이 정의의 일부로 표현된다.

### 전제 조건

- 강제성 축이 사용자의 사고 흐름에서 자연스럽게 작동한다: "이 약속은 항상 지켜야 한다"와 "이 형태가 권장된다"의 차이를 사용자가 인식한다
- 0041에서 principle 엔티티가 폐기되어, rule이 "원칙의 영역적 발현"이라는 제약에서 자유로워진다: rule의 진입에 원칙 도출이 필수 조건이 아니게 된다

### 제약 조건

- user scope의 13개 항목은 그대로 보존된다: 사용자의 자산이며 0037 분화 시의 분류가 유지된다
  - 제약이 없었다면: 셋 → 둘 전환에 맞춰 user scope의 분류도 재정렬했을 것

## 대안

### 단일 엔티티 + 강제성 메타데이터

- 설명: rule과 convention을 단일 엔티티로 합치고, 각 항목에 must/prefer 메타데이터로 강제성을 표현
- 트레이드오프: 어휘적 정체성이 약해진다. 항목을 볼 때마다 메타데이터를 확인해야 무게가 전달되고, 같은 엔티티 안에 무게가 다른 항목이 섞인다

### convention 영역만 확장, rule은 0037 정의 유지

- 설명: convention의 영역만 표현·구조·도구로 확장하고, rule은 "원칙의 영역적 발현" 정의를 그대로 유지
- 트레이드오프: rule과 convention의 경계 모호성이 해소되지 않는다. 사용자 사례 중 다수가 "원칙의 영역적 발현"이 아니지만 강제적이어야 하는 항목이라, rule의 자리에 들어가지 못하고 convention으로 떠밀린다
