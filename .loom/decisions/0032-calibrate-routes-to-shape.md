# calibrate는 concept/decision을 직접 수정하지 않고 shape 재진입으로 되돌린다

## 맥락

calibrate는 기존에 concept↔코드 정합성 감사 시 "교정 제안"을 포함한 동작을 해왔다. 이는 경우에 따라 직접 수정으로 이어질 수 있었다.

calibrate가 직접 수정하면 설계 재진입 없이 구조가 바뀐다. 재설계는 shape의 책임이며, 재진입 경로를 거치지 않으면 대안 탐색과 영향 분석이 누락된 채 변경이 이뤄진다. 또한 "재조정이 필요하다"는 감지와 "재조정을 어떻게 한다"는 설계가 분리되지 않으면, calibrate가 두 역할을 겸하면서 스킬의 경계가 흐려진다.

### 관련 원칙

- **RFP (Redesign From First Principles)**: 변경을 기존 설계에 덧붙이지 않는다. 처음부터 이 요구사항을 알았다면 어떻게 만들었을지 재설계한다. 재설계는 shape의 책임이다.
- **DCB (Draw Clear Boundaries)**: 경계를 분명히 한다. 모듈이 변경되는 이유가 하나가 되도록 관심사를 분리한다. calibrate는 감지, shape는 재설계.

## 결정

calibrate는 concept와 decision을 직접 수정하지 않는다. 정합성 괴리나 재설계 필요성을 감지하면, 그 근거와 함께 shape 재진입 판단을 사용자에게 요청한다.

- **calibrate의 책임**: 전체 범위 감사, 괴리 감지, 재진입 필요성 판단
- **shape의 책임**: 재진입 후 대안 탐색, 재설계, concept/decision 변경

calibrate가 감지한 신호가 shape의 재진입 조건이 된다. 이로써 감지와 재설계가 분리되고, 각 스킬의 변경 이유가 하나로 유지된다.

### 제약 조건

- calibrate는 전체 범위 감사 스킬이다: 직접 수정 책임을 추가하면 감사와 설계가 같은 스킬에 섞인다
- 재설계는 shape에서 이뤄져야 대안 탐색이 보장된다: calibrate에서 즉시 수정하면 EDS의 대안 탐색이 누락된다

### 전제 조건

- shape가 재설계 책임을 맡는다: calibrate가 되돌린 감지 신호를 처리할 수 있다
- calibrate의 감지 결과가 사용자에게 전달된다: 유지 여부와 shape 재진입 여부는 사용자가 판단한다

## 대안

### calibrate가 직접 수정 허용

- 설명: calibrate가 감지와 수정을 함께 수행
- 트레이드오프: 대안 탐색 없이 수정이 이뤄진다. 감지와 재설계가 같은 스킬에 섞여 변경 이유가 둘 이상이 된다

### calibrate가 수정안을 만들고 사용자가 승인

- 설명: calibrate가 구체적 수정안을 만들고 사용자가 승인하면 적용
- 트레이드오프: 수정안 생산이 재설계 과정이다. 결국 shape의 책임을 calibrate에 중첩시키는 구조가 된다

## 영향

- calibrate - 직접 수정 금지, shape 재진입 판단으로 전환
- shape - calibrate의 감지 신호를 재진입 조건으로 수용
- [[concept]] - 변경 경로가 shape로 단일화
- [[decision]] - 변경 경로가 shape로 단일화
