# Principle을 원칙·룰·컨벤션 셋의 엔티티로 분화하되 폴더는 공유한다

## 맥락

principle concept이 단일 엔티티로 다양한 무게감을 흡수하고 있었다. user scope의 13개 항목을 분류해 보면, 사고 양식의 기저(원칙), 영역적 조건-행동 매핑(룰), 합의된 표현의 통일(컨벤션) 세 형태가 섞여 있었다.

세 형태의 무게감과 진입 기준이 본질적으로 다르다. 원칙은 도메인을 넘는 사고 기저이고, 룰은 영역적 패턴이며, 컨벤션은 임의 선택의 합의된 통일이다. 단일 엔티티로 묶을 무게감 정의가 셋에 모두 적용되지 않는다.

또한 auto memory에 직접 피드백·코드 스타일·영역적 패턴이 혼재되어 있는 원인이, 원칙의 무게감에 미달하는 항목들이 갈 자리가 없기 때문이라는 것이 드러났다.

## 결정

원칙·룰·컨벤션을 셋의 도메인 엔티티로 분화한다. 각자 자기 정체성·무게감·진입 기준을 가진다.

폴더는 `principles/` 단일을 유지한다. 폴더 이름은 셋의 대표 어휘로 작동하며, 인덱스 문서에서 layer별 섹션으로 분류를 표현한다.

### 전제 조건

- 1·2·3층 위계가 사용자의 사고 흐름에서 자연스럽게 분리된다: 원칙(사고 기저), 룰(원칙의 영역적 발현), 컨벤션(코드 표현의 합의)이 위계로 이어진다
- 셋이 다른 무게감을 가진다: 진입 기준의 본질이 다름 — 원칙은 당위성 깊게, 룰은 영역의 명료성 추가, 컨벤션은 합의성 핵심

### 제약 조건

- 폴더 이름은 하위 호환을 유지해야 한다: 기존 사용자의 13개 데이터가 그대로 자기 자리에 있어야 한다
  - 제약이 없었다면: 폴더 이름을 셋의 통칭(예: guidelines)으로 변경하여 어휘 충돌을 더 명확히 해소했을 것

## 대안

### 단일 엔티티 + 인덱스 분류

- 설명: principle 도메인 엔티티는 하나로 두고, 인덱스에서 원칙·룰·컨벤션 분류만 표시
- 트레이드오프: 같은 무게감을 셋이 공유한다는 가정이 필요한데 실제로는 다름. 단일 무게감 정의가 셋 모두에 적용되지 않는다.

### 엔티티 분화 + 폴더 분리

- 설명: 셋의 엔티티를 별도 폴더(`principles/`, `rules/`, `conventions/`)에 보관
- 트레이드오프: 폴더 분리의 변경 비용이 크다. 항목의 layer 이동 시 파일 이동·인덱스 갱신·그래프 갱신 모두 필요. 기존 13개 데이터에도 이동 비용 발생.

## 영향

- [[principle]] - 도메인 전체 정의에서 "원칙" layer 정의로 좁아진다
- [[rule]] - 신규 도메인 엔티티
- [[convention]] - 신규 도메인 엔티티
- [[index]] - 도메인 엔티티 목록과 관계 갱신
