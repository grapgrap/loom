# loom의 영역을 실행까지 확장한다

## 맥락

loom은 shape → plan → review 흐름에서 판단 인프라로 기능해왔다. plan과 review 사이의 실행 구간에서 loom의 역할이 부재했고, 이 갭은 "plan-review 사이 추적 가능성 보류"로 열린 질문이었다.

반복 관찰된 문제: plan의 태스크가 실행하기에 충분히 세분화되지 않았을 때, 에이전트가 자의적으로 구현하거나, 사용자가 별도 합의 없이 실행에 돌입하는 상황이 발생했다. decision 0013에서 "shape로 해소한다"고 했지만, 이는 컨셉 해상도 문제에 대한 답이었고 구현 수준의 합의를 다루지 못했다.

## 결정

task 스킬을 도입하여 loom의 영역을 실행까지 확장한다. "loom은 판단 인프라이지 실행 인프라가 아니다"라는 기존 경계를 의도적으로 넓힌다.

task는 plan의 태스크를 입력으로 받아, 방향성 합의 → 코드 초안 → 구현 완료의 단계를 거친다. 이 과정에서 사용자와의 충분한 실행 합의를 이끌어내는 것이 핵심이다.

### 제약 조건

- decision 0013은 여전히 유효하다: 컨셉 해상도 문제는 shape로 해소한다. task는 구현 수준의 합의를 담당한다
- task의 step 1에서 컨셉의 방향성에 큰 변경이 발견되면 shape로 라우팅한다

### 전제 조건

- plan과 실행 사이에 합의 과정이 부재하면 에이전트의 자의적 판단이 반복된다: task가 이 구간을 명시적으로 다룬다
- 구현 과정에서 세밀한 컨셉 변경이 발생할 수 있다: task가 shape와 동일한 산출물 체계를 사용하여 이를 기록한다

## 대안

### shape로 해소 유지 (decision 0013)

- 설명: 실행 전 해상도가 부족하면 shape를 호출하여 컨셉 수준에서 해소한다
- 트레이드오프: 컨셉 해상도에는 유효하나, 구현 수준의 방향성 합의와 코드 초안 협의를 다루지 못한다

### 실행 구간을 loom 밖에 둠

- 설명: loom은 판단 인프라로 남기고, 실행은 사용자와 에이전트의 자율에 맡긴다
- 트레이드오프: plan-review 사이 갭이 지속되며, 실행 합의 부재로 인한 자의적 구현이 반복된다

## 영향

- plan - plan의 태스크가 task 스킬에 의해 소비되고 갱신되는 생명주기를 갖게 된다
