# 다섯 원칙을 한 스킬에 전속시키지 않고 shape/review/calibrate에 분산 배치하여 다중 관문으로 검증한다

## 맥락

다섯 원칙을 스킬의 Step과 판단 지점으로 녹이는 방향이 합의된 후, 다음 질문은 "어떤 원칙을 어느 스킬에 녹일 것인가"였다.

사용자의 관찰 — "원칙들은 shape 단계에서 애초에 잘 지켜지면 좋겠지만, 경험상 에이전트와의 대화에서 흐름에 따라 이야기가 진행되다보면 잘 지켜지지 않는 경우가 많이 있었습니다. 그것을 위해 review와 calibrate가 있는 것입니다." 단일 지점의 완벽성보다 여러 지점의 재검증이 견고하다는 판단이 있었다.

### 관련 원칙

- **OOE (Outcome-Oriented Execution)**: 최종 상태를 먼저 확정하고 역방향으로 트레이드오프한다. 점진적으로 쌓아올리지 않는다.
- **RFP (Redesign From First Principles)**: 변경을 기존 설계에 덧붙이지 않는다. 처음부터 이 요구사항을 알았다면 어떻게 만들었을지 재설계한다.
- **FT (Foundational Thinking)**: 구조적 결정은 옵션 가치를 극대화하고, 코드 레벨 결정은 단순함을 추구한다. 조기 추상화를 경계한다.
- **DCB (Draw Clear Boundaries)**: 경계를 분명히 한다. 모듈이 변경되는 이유가 하나가 되도록 관심사를 분리한다.
- **EDS (Exhaust the Design Space)**: 전례 없는 설계 결정 앞에서 대안을 구체적으로 탐색한 뒤 커밋한다.

## 결정

다섯 원칙 중 어느 것도 한 스킬에 전속되지 않는다. 같은 원칙이 shape(설계 시점) / review(세션 범위 검증) / calibrate(전체 범위 검증)에서 시점과 범위를 달리해 재차 검증되도록 분산한다.

각 원칙의 견제 경로:

- **OOE**: shape(최종 상태 확정) ↔ review(세션 내 제약 변화 감지) ↔ calibrate(누적 제약 변화 감지)
- **RFP**: shape(재설계 판단) ↔ review(세션 내 전제 변화 감지) ↔ calibrate(누적 전제 변화 감지)
- **FT**: shape(조기 분화 경계) ↔ review(성급한 분화 감지) ↔ calibrate(축적된 분화 평가)
- **DCB**: shape(경계 판단) ↔ review(concept 왜곡 감지) ↔ calibrate(누적 경계 드리프트)
- **EDS**: shape(대안 탐색) ↔ review(대안 고려 여부 확인)

review의 세션 범위 검증과 calibrate의 전체 범위 검증은 검증 범위가 다르다. review에서 보이지 않는 누적 드리프트는 calibrate의 전체 관점에서만 드러난다.

### 제약 조건

- 에이전트는 대화 흐름에서 원칙을 놓칠 수 있다: 단일 관문으로는 복원이 보장되지 않는다
- review는 세션 단위로 작동하여 누적 드리프트를 보지 못한다: calibrate가 전체 범위에서 보완한다

### 전제 조건

- 같은 원칙이라도 시점(설계 / 완료 후)과 범위(세션 / 전체)가 다르면 다른 측면이 드러난다
- calibrate가 전체 관점 감사를 수행하도록 설계되어 있다

## 대안

### 한 원칙에 한 스킬 전속 배정

- 설명: 예를 들어 OOE는 shape에만, DCB는 review에만 녹여 단순한 대응 관계 유지
- 트레이드오프: 단일 지점에서 놓치면 복원 경로가 없다. 에이전트 대화 흐름의 실제 패턴과 맞지 않는다

### 원칙 검증을 calibrate에 전담

- 설명: shape/review는 원칙을 내재화하지 않고, 모든 원칙 검증을 calibrate에 집중
- 트레이드오프: 설계와 세션 단계에서 원칙이 유도되지 않으므로, calibrate 이전에 드리프트가 누적된다

- [[review]] - 세션 범위에서 다섯 원칙을 재검증한다
