# 문서 탐색 도구는 선택적·교체 가능하게 제공한다

## 맥락

loom의 문서들은 마크다운에 위키링크를 사용하여 관계를 표현한다. 에이전트가 맥락을 로드할 때 이 관계를 탐색해야 하는데, 현재는 grep으로 수행하고 있다. grep은 매칭 라인의 원문을 전부 반환하므로 토큰 경제성이 낮다.

문서 탐색을 효율화할 도구로 Obsidian CLI와 qmd를 검토했다.

- **Obsidian CLI**: 위키링크 구조 탐색(links, backlinks, orphans, unresolved)에 최적이고 토큰 효율이 가장 높다. 단, Obsidian 앱이 실행 중이어야 하며 vault 등록이 필요하다.
- **qmd**: BM25 키워드 검색과 벡터 의미 검색을 제공하며, MCP 서버로 에이전트와 네이티브 통합된다. 독립 실행이 가능하다. 의미 검색은 ~2GB 모델 다운로드가 필요하다.

두 도구의 관심사가 다르다 — Obsidian CLI는 구조적 탐색, qmd는 의미적 검색. loom을 사용하는 프로젝트의 환경이 다양하므로, 특정 도구를 강제할 수 없다.

## 결정

문서 탐색 도구를 선택적으로 제공하며, 사용자가 백엔드를 선택할 수 있게 한다.

- 문서 탐색은 선택적 기능이다 — 도구가 없어도 loom은 동작한다
- 사용자가 백엔드를 선택한다 — Obsidian CLI, qmd, 혹은 둘 다
- 도구가 없는 환경에서는 grep으로 폴백한다

### 제약 조건

- loom이 특정 도구에 의존하면 채택 장벽이 높아진다: 도구는 선택적이어야 한다

### 전제 조건

- 프로젝트 규모가 커질수록 토큰 효율적인 문서 탐색이 필요해진다: 도구 도입의 가치가 있다
- 두 도구의 관심사가 겹치지 않는다: 둘 다 채용하는 것이 가능하다

## 대안

### Obsidian CLI만 채용

- 설명: 구조적 탐색에 최적이고 토큰 효율이 가장 높다
- 트레이드오프: Obsidian 앱 의존성이 플러그인 배포와 맞지 않는다

### qmd만 채용

- 설명: 독립 실행 가능하고 MCP 통합이 네이티브이다
- 트레이드오프: 위키링크 구조를 이해하지 못하며, 의미 검색에 ~2GB 모델이 필요하다

### 도구 없이 grep으로 유지

- 설명: 추가 의존성 없이 현재 방식을 유지한다
- 트레이드오프: 문서가 늘어날수록 토큰 소비가 선형 증가한다
