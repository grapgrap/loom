# Scope

여러 `.loom/` 계층이 존재할 때, 엔티티 간 우선순위를 결정하는 체계.

## 목적

하나의 작업 맥락에서 여러 `.loom/` 디렉토리가 존재할 때, 엔티티를 어떤 순서로 로드하고 어떤 것이 우선하는지를 정의한다.

### 핵심 시나리오

- 모노레포에서 패키지별 `.loom/`이 있으면, 해당 패키지의 원칙이 루트의 원칙보다 우선한다
- 개인 설계 원칙을 `~/.claude/.loom/`에 두면, 어떤 프로젝트에서든 base로 적용된다

## 동작 원리

### 계층

두 종류의 스코프가 존재한다:

- **user**: `~/.claude/.loom/` — 개인의 프로젝트 독립적 엔티티
- **project**: `.loom/` — 디렉토리 계층 어디에든 존재할 수 있으며, 중첩 가능

### 해소 규칙

사용자의 발화에서 작업 대상의 맥락을 읽고 해당 스코프를 식별한다. 스코프 식별은 cwd가 아니라 발화의 맥락에 의존한다. 모노레포에서 사용자는 프로젝트 루트에서 하위 프로젝트를 접근하므로, cwd는 스코프의 신뢰할 수 있는 기준이 아니다.

**가장 가까운 스코프가 이긴다.** 동일한 판단 상황에서 더 구체적인(작업 대상에 가까운) 엔티티가 우선한다.

로드 순서:
1. 식별된 scope의 `.loom/`에서 엔티티를 읽는다
2. cwd까지 상위 디렉토리에 `.loom/`이 있으면 병합한다 (중첩 시 반복)
3. 마지막에 `~/.claude/.loom/`을 base로 붙인다 (principle만 해당)

### 엔티티별 유효 스코프

| 엔티티 | user | project |
|--------|------|---------|
| principle | O | O |
| concept | X | O |
| decision | X | O |
| plan | X | O |

principle만 user와 project를 관통한다. concept, decision, plan은 실행 맥락이 필요하므로 project scope에서만 존재한다.

## 트레이드 오프

- 스코프 간 충돌을 메타데이터나 파일명 매칭으로 해소하지 않는다 — 에이전트가 내용을 이해하고 판단한다
- 스코프 탐색 깊이에 상한을 두지 않는다 — 디렉토리 구조가 곧 스코프 구조다
