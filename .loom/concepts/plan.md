# Plan

설계를 실행 가능한 태스크와 의존관계로 구조화한 문서.

## 목적

무엇을 어떤 순서로 할 것인지를 명시하여, 구현의 실행과 진행 추적을 가능하게 한다.

## 책임

요구사항을 독립적으로 실행 가능한 태스크로 분해하고, 태스크 간 선후관계와 병렬 가능성(의존관계)을 명시하며, 각 태스크의 거시적 실행 가능성을 사용자와 함께 검증하고, 진행 상황을 체크리스트로 추적한다.

## 정책

- 산출물이지 실행이 아니다 — plan 자체는 코드를 만들지 않는다.
- concept·flow·decision과 연관되지만 독립적으로 존재하며, 독자적 생명주기를 갖는다 — 엔티티의 수명과 결합되지 않는다.
- 하나의 concept이나 flow에서 여러 plan이 나올 수 있다.
- 태스크는 작성 시점의 해상도를 반영한다 — 초기 plan이 거친 것은 자연스러운 일이다.
- 태스크는 task 스킬에 의해 소비되고 완료 처리된다 — plan은 산출물을 내면 역할이 종료되지만, 태스크의 진행 상태는 task가 갱신한다.
- 초안의 확정 전에 각 태스크를 사용자와 함께 검증한다 — 사용자가 변경 범위를 인지하고 그 정당성을 판단할 수 있어야 하므로, 에이전트 단독 점검으로 대체하지 않는다.
- 검증의 깊이는 각 태스크가 닿을 거점이 실재하고 접근 가능한지를 확인하는 수준에 머문다 — 더 깊은 해상도가 필요한 태스크는 파고들지 않고 shape나 task로 보낼 표식만 남긴다.

## 경계

- 단일 작업이나 기계적 변환에는 plan을 작성하지 않는다 — shape의 완료 후 라우팅에서 판단한다.
- 태스크의 코드 수준 실행 충분성은 plan의 책임 밖이다 — plan의 검증은 거점의 실재와 의존의 연결 같은 거시적 실행 가능성까지만 본다. 깊은 해상도가 필요하면 shape로, 구현 수준의 합의가 필요하면 task로 해소한다.
