# Plan

설계를 실행 가능한 태스크와 의존관계로 구조화하고, 작성과 실행에 걸쳐 갱신되는 살아있는 문서.

## 목적

무엇을 어떤 순서로 할 것인지를 명시하여 구현의 실행과 진행 추적을 가능하게 하고, 세션을 가로질러 작업 맥락이 누락 없이 전달되게 한다.

## 책임

요구사항을 독립적으로 실행 가능한 태스크로 분해하고, 태스크 간 선후관계와 병렬 가능성(의존관계)을 명시하며, 각 태스크의 거시적 실행 가능성을 사용자와 함께 검증한다. 실행 단계에서는 진행 맥락 — 진행 상태, 다음으로의 핸드오프, 작업 중 알게 된 사실 — 을 누적하여 세션 간 전달을 담보한다.

## 정책

- 산출물이지 실행이 아니다 — plan 자체는 코드를 만들지 않는다.
- concept·flow·decision과 연관되지만 독립적으로 존재하며, 독자적 생명주기를 갖는다 — 엔티티의 수명과 결합되지 않는다.
- 하나의 concept이나 flow에서 여러 plan이 나올 수 있다.
- 태스크는 작성 시점의 해상도를 반영한다 — 초기 plan이 거친 것은 자연스러운 일이다.
- plan은 작성으로 끝나지 않고 task의 실행 내내 갱신되는 살아있는 문서다 — task가 진행 맥락(진행 상태, 핸드오프, 작업 중 알게 된 사실)을 누적하고, 가정이 현실과 어긋나면 plan을 맹신하지 않고 남은 태스크와 순서를 갱신한다.
- 초안의 확정 전에 각 태스크를 사용자와 함께 검증한다 — 사용자가 변경 범위를 인지하고 그 정당성을 판단할 수 있어야 하므로, 에이전트 단독 점검으로 대체하지 않는다.
- 검증의 깊이는 각 태스크가 닿을 거점이 실재하고 접근 가능한지를 확인하는 수준에 머문다 — 더 깊은 해상도가 필요한 태스크는 파고들지 않고 shape나 task로 보낼 표식만 남긴다.

## 경계

- 단일 작업이나 기계적 변환에는 plan을 작성하지 않는다 — shape의 완료 후 라우팅에서 판단한다.
- 태스크의 코드 수준 실행 충분성은 plan의 책임 밖이다 — plan의 검증은 거점의 실재와 의존의 연결 같은 거시적 실행 가능성까지만 본다. 깊은 해상도가 필요하면 shape로, 구현 수준의 합의가 필요하면 task로 해소한다.
- 진행 맥락은 담되 설계 의도는 담지 않는다 — 왜 그렇게 결정했는가는 [[decision]]의 몫이다. plan은 지금 상태가 어떠하며 다음이 무엇을 알아야 하는가를 운반한다.

## 관계

- [[concept]] - concept을 참조하여 범위를 한정한다.
- [[flow]] - flow를 참조하여 구현 순서의 단서를 얻는다.
- [[decision]] - decision을 참조하여 설계 의도와 제약을 이해한다. 진행 맥락(plan)과 설계 의도(decision)의 경계를 공유한다.
- [[rule]] - rule을 자동 참조하여 태스크 구성에 반영한다.
- [[convention]] - convention을 자동 참조하여 태스크 구성에 반영한다.
