# Rule

이 프로젝트에서 항상 지켜져야 하는 비기능 요구사항.

## 목적

비기능 요구사항을 명시화하여, shape·plan·task가 자동으로 참조해 일관된 설계와 코드를 산출하게 한다. 코드 작성 위임 시점에 반복되는 피드백을 제거하는 것이 핵심 동기다.

## 책임

이 프로젝트의 비기능 요구사항을 명시하고, 도구 카탈로그 모델로 각 스킬이 활용 시점에 골라 적용하게 한다. 각 스킬은 룰 카탈로그 로드 단계에서 인덱스로 카탈로그를 인지하고, 수행하려는 행동에 적용될 항목을 골라, 본문을 로드해 적용 방식을 파악한 뒤 행동한다 - 인덱스의 한 줄만으로 활용을 결정하지 않는다.

## 정책

- 세 가지 조건을 충족해야 한다 - 반복성(같은 행동이 반복적으로 정당화되는가), 당위성(근거가 명확한가), 강제성(위반 시 받아들이지 않는 무게인가).
- 정의에 적용 범위(영역적·상황적 조건)를 명시한다 - 적용되지 않는 곳에서는 무관하다.
- 강제성 축의 무거운 자리이며, 합의된 순간부터 적용 범위 안의 모든 산출물에 적용된다.
- 깊이가 다양한 항목이 공존한다 - 작동 시점 메타데이터는 두지 않고, 깊이는 정의의 추상화 수준에서 드러난다.
- 변경 근거는 decision으로 기록한다 - 누적된 decision이 새 rule의 판단 기준이 된다.
- 사용자의 것이다 - 시스템이 임의로 삭제하지 않으며, project scope의 `.loom/rules/`에 보관된다.

## 경계

- 강제성이 약하고 대체가 허용되는 합의는 convention의 영역이다 - rule이 아니다.
- 프로젝트 횡단의 설계 판단(원칙)은 rule이 아니다 - 사용자 영역(머릿속, user scope, 스킬 본문에 녹은 형태)에 머문다.
- 깊은 판단 근거는 별도 엔티티로 정제·보관하지 않는다 - decision의 누적이 그 역할을 한다.

## 관계

- [[convention]] - 강제성 축의 다른 극 (대체 가능한 형태).
- [[decision]] - rule의 근거와 변경 맥락이 기록된다.
- [[concept]] · [[flow]] - 이들이 다루는 도메인과 흐름 위에서 rule이 작동한다.
- shape·plan·task - 산출물(concept·flow, 태스크, 코드)을 rule을 만족하는 형태로 만든다.
- review·calibrate - 구현이 rule을 만족하는지 검증하고, rule 자체를 누적 점검한다.
