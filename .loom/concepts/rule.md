# Rule

이 프로젝트에서 항상 지켜져야 하는 비기능 요구사항.

## 목적

이 프로젝트의 비기능 요구사항을 명시화하여, shape·plan·task가 자동으로 참조하여 일관된 설계와 코드를 산출하게 한다. 코드 작성 위임 시점에 반복되는 피드백을 제거하는 것이 핵심 동기다.

### 핵심 시나리오

- 스킬이 rule을 자동으로 읽어 적용 범위에 들어가는 항목을 만족하는 산출물을 만든다
- 비기능 요구사항 중 강제 사항이 발생할 때 rule로 명시한다
- 영역적 또는 상황적 조건이 충족되는 범위에서 일관된 행동이 필요할 때 — 조건을 정의에 포함하여 표현한다

## 동작 원리

### 정체성

rule은 "이 프로젝트에서 지켜야 할 비기능 요구사항"의 형태를 가진다.
강제성 축의 무거운 자리이며, 합의된 순간부터 적용 범위 안의 모든 산출물에 적용된다.

프로젝트 종속이다 — 같은 항목이 다른 프로젝트에서는 rule이 아닐 수 있다. 프로젝트 횡단의 설계 판단(원칙)은 rule이 아니며, 사용자 본인의 영역(머릿속, user scope, 또는 스킬 본문에 녹은 형태)에 머문다.

적용 범위는 영역적 조건이나 상황적 조건으로 표현할 수 있다. 조건이 충족되는 범위 안에서는 강제력을 가진다.

rule 안에 깊이가 다양한 항목이 공존한다. 작동 시점 메타데이터는 두지 않으며, 깊이는 정의 자체의 추상화 수준에서 자연스럽게 드러난다.

### 핵심 동작

- 세 가지 조건을 충족해야 한다:
  1. **반복성**: 해당 영역에서 반복적으로 같은 행동이 정당화되는가
  2. **당위성**: 그 행동의 근거가 충분히 명확한가
  3. **강제성**: 위반 시 받아들이지 않는 무게를 가지는가
- 정의에 적용 범위(영역적 또는 상황적 조건)를 명시한다 — 적용되지 않는 곳에서는 무관하다
- 변경의 근거는 decision으로 기록한다 — 누적된 decision이 새 rule의 판단 기준 역할을 한다
- 사용자의 것이다 — 시스템이 임의로 삭제하지 않는다
- project scope의 `.loom/rules/`에 보관된다 — 각 스킬이 룰 카탈로그 로드 단계에서 인덱스를 통해 인지한다

### 활용

rule은 도구 카탈로그 모델로 활용된다. 각 스킬은 다음 흐름을 따른다:

1. **카탈로그 인지**: 룰 카탈로그 로드 단계에서 인덱스를 통해 어떤 rule이 있는지 인지한다
2. **도구 선택**: 활용 시점에 수행하려는 행동을 의식적으로 명시하고, 카탈로그에서 그 행동에 적용될 가능성 있는 항목을 골라낸다
3. **본문 로드**: 골라낸 rule의 본문 파일을 로드해 정확한 적용 방식(정의, 적용 범위)을 파악한다
4. **행동**: 본문에 따라 행동한다. 인덱스의 한 줄만으로 활용 결정을 내리지 않는다

각 스킬의 자기 산출물 관점:

- **shape**: rule이 새 concept에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 설계한다. 충돌 시 사용자에게 보고
- **plan**: rule이 태스크에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 태스크를 분해
- **task**: rule이 코드에 적용되는지 식별하고, 적용된다면 rule을 만족하는 형태로 코드를 작성
- **review**: 구현이 적용되는 rule을 만족하는지 사후 검증
- **calibrate**: rule 자체를 누적 점검하고 신규 항목을 발견

작동 시점이 다양한 rule은 자연스럽게 여러 스킬에서 활성화된다.

## 트레이드 오프

- 강제성이 약하고 대체가 허용되는 합의는 convention의 영역이다 — rule이 아니다
- 프로젝트 횡단의 설계 판단은 rule이 아니다 — 사용자 영역(머릿속, user scope, 또는 스킬 본문에 녹은 형태)에 머문다
- 영역이 모호한 rule은 적용 범위를 정의에 명시하여 한정한다 — 무엇에 적용되고 무엇에 적용되지 않는지 분명해야 한다
- rule의 깊은 판단 근거는 별도 엔티티로 정제·보관하지 않는다 — 변경 근거가 decision으로 기록되고, 누적된 decision이 그 역할을 한다

## 관련 컨셉

- [[convention]] - 강제성 축의 다른 극 (선호되는 형태)
- [[decision]] - rule의 근거와 변경 맥락이 기록된다
- [[concept]] - concept이 다루는 도메인 위에서 rule이 작동한다
- shape - 새 concept을 rule을 만족하는 형태로 설계한다
- plan - 태스크를 rule을 만족하는 형태로 분해한다
- task - 코드를 rule을 만족하는 형태로 작성한다
- review - 구현이 적용 범위 내 rule을 만족하는지 사후 검증한다
- calibrate - rule 자체를 누적 점검하고 신규 항목을 발견한다
