# Convention

강하게 권장되지만 대체 가능한 형태.

## 목적

프로젝트에서 합의된 표현·구조·도구의 통일을 명시화하여, 코드 작성 위임 시 일관성을 보장한다. rule과 같은 자동 참조 경로를 공유하지만, 강제성이 약해 정당한 사유 아래 대체가 허용된다.

## 책임

표현·구조·도구의 합의된 통일을 명시하고, rule과 같은 도구 카탈로그 모델로 각 스킬이 활용 시점에 골라 적용하게 한다. 명명 규칙뿐 아니라 모듈 구성 패턴, 선호 라이브러리도 포함한다.

## 정책

- 두 가지 조건을 충족해야 한다 - 반복성(형태가 반복적으로 등장하는가), 합의성(어느 형태로 통일되어야 한다는 합의가 형성되는가).
- 도구·언어·맥락에 종속적이다 - 같은 형태가 다른 환경에서 다른 convention이 될 수 있다.
- 정당한 사유가 있다면 합의된 형태 외의 선택이 허용된다 - 다만 사유 없는 일탈은 허용하지 않으며, 합의된 형태 안에서는 일관성이 보호된다.
- 사용자의 것이다 - 시스템이 임의로 삭제하지 않으며, project scope의 `.loom/rules/`에 보관된다.

## 경계

- 위반 시 받아들이지 않는 무게의 합의는 rule의 영역이다 — convention이 아니다.
- 일회성 합의는 아직 convention이 아니다 — 반복이 확인될 때 축적한다.

## 관계

- [[rule]] - 강제성 축의 다른 극 (항상 지켜야 하는 강제적 합의).
- [[decision]] - convention의 근거와 변경 맥락이 기록된다.
- [[concept]] · [[flow]] - 이들이 다루는 도메인과 흐름 위에서 convention이 작동한다.
- plan·task - 산출물을 convention의 형태로 만든다.
- review·calibrate - 구현이 convention과 일치하는지 검증하고, convention 자체를 누적 점검한다.
