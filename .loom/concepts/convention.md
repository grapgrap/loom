# Convention

강하게 권장되지만 대체 가능한 형태.

## 목적

프로젝트에서 합의된 표현·구조·도구의 통일을 명시화하여, 코드 작성 위임 시 일관성을 보장한다. rule과 같은 자동 참조 경로를 공유하지만, 강제성이 약하여 정당한 사유 아래 대체가 허용된다.

### 핵심 시나리오

- 코드 표현의 일관성이 필요할 때 (이름 규칙, 들여쓰기 등)
- 구조 패턴의 통일이 필요할 때 (모듈 구성, 함수 시그니처 형태 등)
- 도구·라이브러리 선택의 통일이 필요할 때 (선호 라이브러리 등)

## 동작 원리

### 정체성

convention은 "이 형태로 통일한다"의 형태를 가진다.
강제성 축에서 rule보다 가벼운 자리이며, 강하게 권장되지만 정당한 사유 아래 대체가 허용된다.

표현·구조·도구의 합의된 통일을 모두 포함한다. 명명규칙뿐 아니라 모듈 구성 패턴, 선호 라이브러리도 convention의 영역이다.

### 핵심 동작

- 두 가지 조건을 충족해야 한다:
  1. **반복성**: 해당 형태가 반복적으로 등장하는가
  2. **합의성**: 어느 형태로 통일되어야 한다는 합의가 형성되는가
- 도구·언어·맥락에 종속적이다 — 같은 형태가 다른 환경에서 다른 convention이 될 수 있다
- 대체 가능성을 가진다 — 정당한 사유가 있다면 합의된 형태 외의 선택이 허용된다
- 사용자의 것이다 — 시스템이 임의로 삭제하지 않는다
- project scope의 `.loom/rules/`에 보관된다 — plan과 task가 Step 0에서 자동 로드한다

## 트레이드 오프

- 위반 시 받아들이지 않는 무게의 합의는 rule의 영역이다 — convention이 아니다
- 일회성 합의는 아직 convention이 아니다 — 반복이 확인될 때 축적한다
- convention은 가벼운 자리이지만 합의된 형태 안에서는 일관성이 보호된다 — 임의 선택과는 다르다

## 관련 컨셉

- [[rule]] - 강제성 축의 다른 극 (항상 지켜야 하는 강제적 합의)
- [[decision]] - convention의 근거와 변경 맥락이 기록된다
- [[concept]] - concept이 다루는 도메인 위에서 convention이 작동한다
- plan - plan이 convention을 자동 참조하여 태스크를 구성한다
