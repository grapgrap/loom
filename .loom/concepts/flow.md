# Flow

도메인 사이를 사건이 가로지르는 협력의 경로를 기술하는 영속 문서.

## 목적

어느 한 노드의 내부로 환원되지 않는 도메인 간 협력의 경로를, 정렬의 공유 맥락으로 남긴다. concept이 전략적 설계 레이어의 노드라면, flow는 같은 레이어의 경로다. 분해(concept)가 기본 축이고, flow는 사건이 도메인 경계를 가로지르는 경로가 실재할 때 출현하는 조건부 축이다.

## 책임

하나의 트리거에서 결과에 이르기까지, 도메인과 도메인 사이를 사건이 가로지르며 상태가 전이되는 연쇄를 단계로 기술한다. 각 단계가 어느 도메인의 관여인지를 드러냄으로써, flow 자체가 도메인들의 경계를 비추는 렌즈가 된다.

## 정책

- 두 관문으로 판별한다 — 단계 사이를 흐르는 것이 상태를 전이시키는 사건이고(사건성), 경로가 도메인 경계를 가로지를 때(경계 횡단성)만 flow로 남긴다. 데이터가 변환되어 넘어갈 뿐인 계산의 연쇄와 한 도메인 내부의 동작은 [[concept]]의 몫이다.
- 명명 가능성은 판별 근거가 아니다 — 조정에 이름이 붙는다는 사실은 그것을 노드로 만들지 않는다. 자기 상태도 규칙도 없는 조정의 이름은 노드의 이름이 아니라 flow의 이름이다.
- 경로는 노드에 귀속되지 않는다 — 자기 상태나 규칙을 지닌 조정자가 있으면 그것은 [[concept]] 노드로서 flow에 등장할 뿐, 경로를 흡수하지 않는다. 노드가 소유하는 것은 상태와 결정이지 경로가 아니다.
- 단계는 도메인 경계를 넘는 마디 단위이며, 한 도메인 내부의 구현은 담지 않는다.
- 현재 상태의 스냅샷으로 동작한다 — 변경의 역사는 [[decision]]이 진다.
- 프랙탈로 합성된다 — 한 단계가 하위 흐름을 가리킬 수 있고, 한 자리에 여러 대안 흐름(변이)이 올 수 있다.

## 경계

- 도메인의 정적 정의는 다루지 않는다 — [[concept]]의 몫이다. 조정의 상태와 규칙도 그것을 지닌 조정자 concept의 몫이다.
- 데이터 변환의 연쇄는 다루지 않는다 — 여러 입력을 받아 한 결과로 닫히는 계산은 [[concept]]의 책임이다.
- 구현의 알고리즘·자료구조·라이브러리는 다루지 않는다 — 코드의 몫이며, 그 설계 판단은 [[decision]]에 남는다. 경로가 코드에서 어떤 형태와 이름으로 실현되는가도 같은 자리다.
- 실행 작업의 분해와 순서는 다루지 않는다 — plan의 몫이다.

## 관계

- [[concept]] - flow가 가로지르는 도메인들. concept이 기본 분해 축(노드)이고, flow는 같은 전략 레이어의 조건부 축(경로)이다.
- [[decision]] - flow의 변경 맥락이 기록된다.
- [[plan]] - 실행 시 flow가 태스크 구성의 참조가 된다.
