# Shape

요구사항을 컨셉과 방향성 수준의 설계로 구체화한다.

shape의 How는 방향성 수준이다 — 접근 전략, 제약, 트레이드오프.
구체적 기술 선택, 파일 구조, 함수 시그니처는 shape의 범위가 아니다.

## Step 0 — Triage

입력의 추상화 수준을 판단한다.

shape는 기능/시스템 단위뿐 아니라 plan 태스크 단위에서도 호출될 수 있다.
태스크의 해상도가 실행에 부족할 때, 에이전트나 사용자가 해당 태스크를 shape에 입력한다.

1. 작업 대상 프로젝트의 `.loom/index.md`를 읽어 도메인 전체 지도를 파악한다.
2. `aeira search -s {프로젝트의 .loom 경로} "{입력 키워드}"` 로 관련 문서를 식별한다.
3. 식별된 문서에서 `aeira graph neighbors -s {프로젝트의 .loom 경로} "{node}"` 로 연결 문서를 탐색한다 (outgoing: 연관 concept, incoming: 참조 decision).
4. 탐색된 concept과 decision을 읽는다.
5. 입력을 다음 기준으로 분류한다:

| 수준               | 신호                                                                                | 행동                                                                                         |
| ------------------ | ----------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| 이미 구체적        | 태스크 단위가 명확하고, 영향 범위가 한정적이며, 어떤 concept을 변경하는지 특정 가능 | **plan으로 라우팅** — "이 입력은 shape 없이 plan으로 진행할 수 있습니다. plan을 시작할까요?" |
| 방향은 있으나 모호 | 무엇을 원하는지는 알지만, 경계나 접근법이 불명확                                    | shape 프로세스 진행                                                                          |
| 비전 수준          | 한 줄의 아이디어, 넓은 목표                                                         | shape 프로세스 진행 (분해부터 시작)                                                          |

plan 태스크에서 진입한 경우, "이미 구체적"은 실행으로 라우팅한다 (plan은 이미 존재하므로).

triage 결과를 사용자에게 알리고 동의를 구한다.

## Step 1 — 룰 카탈로그 로드

`.loom/rules/index.md`를 읽어 어떤 룰과 컨벤션이 있는지 카탈로그를 인지한다.

rule은 이 프로젝트에서 항상 지켜져야 하는 강제적 합의, convention은 강하게 권장되지만 대체 가능한 형태이다. 인덱스를 통해 항목 목록과 각 항목의 호출 상황을 파악한다. 본문은 활용 시점에 필요한 항목을 골라낸 뒤 로드한다.

## Step 2 — 요구사항 구체화

입력의 요구사항을 대화로 구체화한다.

- 달성하고자 하는 것은 무엇인가?
- 어떤 상호작용이 일어나는가? 누가 언제 무엇을 하는가?
- 어떤 제약이 있는가? 반드시 지켜야 할 조건은?

이 단계의 초점은 요구사항 자체의 구체화이다. concept이 자연스럽게 언급될 수 있으나, 구체적 파일 구조, 함수 시그니처, 라이브러리 선택 같은 구현 수준 질문은 삼간다. 요구사항이 충분히 구체화되면 Step 3으로 넘어간다.

plan 태스크 단위로 shape가 호출된 경우(decision 0013), 태스크의 문맥(어떤 concept에서 파생되었는지, 의존관계)이 요구사항의 맥락이 된다. 이 경우 Step 2는 태스크 해상도 부족의 원인을 식별하는 역할을 한다.

## Step 3 — 엔티티와 경계 수립

구체화된 요구사항에서 도메인 엔티티를 수집하고, 그 경계를 드러내는 과정을 반복한다.

### 수집과 경계의 반복 사이클

1. **수집**: 요구사항에서 도메인 엔티티를 수집한다.

2. **경계 드러냄**: 수집된 엔티티 간의 책임 경계를 긋는다.
   - 각 엔티티의 변경 이유가 하나가 되는가?
   - 경계가 일상 개념이나 인터페이스 기대와 정합하는가?
   - 전례 없는 설계에서는 대안을 구체적으로 꺼내 본 뒤 사용자와 합의한다.

3. **빈 부분 발견 시 재수집**: 경계를 긋는 과정에서 책임이 비어 있는 지점이나 엔티티 간 관계의 빈틈이 보이면 수집으로 돌아간다.

4. **반복 종료**: 사용자가 구현 수준 이야기(파일 구조, API 시그니처, 구체적 라이브러리 등)를 시작하면 반복이 충분한 신호이다. Step 4로 넘어간다.

### 감지 관점 (반복 전 과정에 상시)

수집과 경계 판단을 하면서 다음 극단에 대한 감지를 유지한다:

- **분화 부족**: 분화해야 하는데 기존 concept에 덧붙이거나(비대화), 예외나 단서로 감싸거나(의미 흐려짐), 삭제해야 하는데 방치(데드 concept)
- **과잉 분화**: 분화가 불필요한데 쪼갬. 이 분화가 지금 실제로 필요한 쓰임을 가지는지 묻는다 — 옵션 가치가 충분하지 않으면 미루는 편이 낫다

### 전제·제약 재평가

수집과 경계 판단을 하면서 기존 concept이나 decision의 전제 또는 제약에 무효화 정황이 드러났는지 수시로 감지한다.

- 무효화가 드러나면 사용자에게 재설계 또는 유지 판단을 요청한다
- 유지 판단은 사용자의 영역이다 — 에이전트는 감지하고 근거를 수집한다
- 유지 선택 시 Step 4에서 "유지 decision"을 생산한다
- 재설계 선택 시 이 Step 안에서 방향을 다시 수립한다

### 룰 카탈로그 활용

수집과 경계 판단 시, 인지한 룰 카탈로그를 도구 선택 절차로 활용한다:

1. 지금 수행하려는 행동을 의식적으로 명시한다
2. 카탈로그에서 그 행동에 적용될 가능성 있는 항목을 골라낸다
3. 골라낸 항목의 본문 파일을 로드해 적용 방식을 파악한다
4. 본문에 따라 설계를 진행한다

적용된 rule을 만족하지 못하는 설계 방향이 발견되면 사용자에게 보고한다.

### 결정 기록

대화 중 결정이 발생하면 즉시 `.loom/decisions/`에 기록한다.
결정의 신호: 대안이 제시되었고, 사용자가 하나를 선택했으며, 그 선택에 근거가 있을 때.
발견 즉시 기록하여 맥락 유실을 방지한다.

## Step 4 — 산출물 작성

### Concept 생성/수정

새로운 도메인 엔티티가 식별되면 `.loom/concepts/`에 파일을 생성한다.
기존 concept이 변경되면 해당 파일을 수정한다.

concept에는 "무엇인가"와 "왜 그런 형태인가"만 기술한다.
시점 종속 항목("아직", "현재", "미정", "후속", "구현 단계에서" 등)을 식별하면 concept이 아닌 시점 종속 문서로 라우팅한다:

- concept의 형태에 영향을 준 보류 → decision에 기록 (보류 결정)
- 나중에 해야 할 작업 → plan의 태스크로 이동

템플릿: `templates/concept.md`를 읽어 사용한다.

### Decision 기록

설계 과정에서 concept이나 구현의 형태에 영향을 준 결정을 `.loom/decisions/`에 기록한다.

기록할 것:

- concept의 구조적 방향을 결정하는 선택 (아키텍처, 접근 전략, 기술 선택)
- 구현(스킬 등)의 동작 범위를 결정하는 선택
- 대안이 존재했고, 선택이 concept이나 구현의 형태에 영향을 준 것
- concept의 범위를 한정하는 보류 결정 (예: "비동기 모델을 지금 결정하지 않는다")
- **유지 decision** — 전제 또는 제약에 무효화 정황이 드러났는데도 기존 concept이나 결정을 유지한 경우. 무효화된 전제·제약, 유지 근거, 유지 결정의 유효 기한을 기록한다. 변화 정황이 없으면 기록하지 않는다

기록하지 않을 것:

- 버그 픽스 — 의도된 동작으로의 복원이므로 설계 결정이 아니다
- plan의 스코프 조정 (예: "이 작업을 후속으로 미룬다")
- 작업 순서나 우선순위
- 자명한 선택 (예: 네이밍)

decision 간 참조 규칙:

- 대체·폐기 참조는 두지 않는다 — 각 decision은 자기완결적이다
- 갱신·계승 참조는 허용한다 (유지 decision이 기존 decision의 전제·제약 변화를 기록하는 경우 등). 참조된 decision을 읽지 않아도 현재 decision의 맥락을 이해할 수 있어야 한다

템플릿: `templates/decision.md`를 읽어 사용한다.

### Concept Index 갱신

concept 문서를 생성/수정한 경우, `.loom/index.md`의 갱신이 필요한지 판단한다.
갱신이 필요한 경우: 새 엔티티 추가, 엔티티 간 관계 변경, 기존 엔티티의 책임 변경.
index는 프로젝트 탐색의 지도 역할을 하므로, 엔티티 목록과 관계도를 최신 상태로 유지한다.

### 파일 이름 규칙

- concept: `{컨셉-이름}.md` (예: `user-authentication.md`)
- decision: `{NNNN}-{결정-제목}.md` (예: `0001-session-based-auth.md`)

decision은 순번을 부여하여 시간순을 보존한다.

### 그래프 갱신

`.loom/` 내 문서를 생성하거나 수정한 후 `aeira sync -s {프로젝트의 .loom 경로}` 를 실행한다.

## Summary

shape 완료 시 다음 형식으로 정리한다:

```
## Shape Summary
- 입력: [원래 요구사항 한 줄 요약]
- Concepts: [생성/수정된 concept 파일 목록]
- Decisions: [기록된 decision 파일 목록]
- 다음 단계: [plan 진행 여부와 범위]
```

## 완료 후 라우팅

shape 완료 시 다음 단계를 판단한다.

| 산출물 상태        | 신호                                                                            | 행동                                                               |
| ------------------ | ------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 태스크 분해가 필요 | 영향 범위가 넓고, 구현 순서에 의존관계가 있으며, 복수의 독립적 작업 단위가 존재 | **plan으로 진행**                                                  |
| 즉시 실행 가능     | shape의 산출물이 단일 작업 또는 기계적 변환이며, 분해할 태스크가 없음           | **직접 실행** — "이 설계는 plan 없이 실행 가능합니다. 진행할까요?" |
| 추가 설계 필요     | shape 과정에서 새로운 모호함이 발견됨                                           | **shape 재진입**                                                   |

라우팅 결과를 사용자에게 알리고 동의를 구한다.
