# Shape

요구사항을 컨셉과 방향성 수준의 설계로 구체화한다.

shape의 How는 방향성 수준이다 — 접근 전략, 제약, 트레이드오프.
구체적 기술 선택, 파일 구조, 함수 시그니처는 shape의 범위가 아니다.

## Step 0 — Triage

입력의 추상화 수준을 판단한다.

1. `.loom/concepts/`를 읽어 기존 도메인 엔티티를 파악한다.
2. `.loom/principles/index.md`를 읽어 원칙을 로드한다.
3. 입력을 다음 기준으로 분류한다:

| 수준 | 신호 | 행동 |
|------|------|------|
| 이미 구체적 | 태스크 단위가 명확하고, 영향 범위가 한정적이며, 어떤 concept을 변경하는지 특정 가능 | **plan으로 라우팅** — "이 입력은 shape 없이 plan으로 진행할 수 있습니다. plan을 시작할까요?" |
| 방향은 있으나 모호 | 무엇을 원하는지는 알지만, 경계나 접근법이 불명확 | shape 프로세스 진행 |
| 비전 수준 | 한 줄의 아이디어, 넓은 목표 | shape 프로세스 진행 (분해부터 시작) |

triage 결과를 사용자에게 알리고 동의를 구한다.

## Step 1 — 도메인 탐색

기존 concept들과 입력의 관계를 파악한다.

- 입력이 기존 concept의 변경인가, 새로운 concept의 도입인가?
- 영향받는 concept들은 무엇인가?
- concept 간 새로운 관계가 생기는가?

기존 concept이 없는 초기 프로젝트라면, 입력에서 도메인 엔티티를 식별하는 것부터 시작한다.

## Step 2 — 설계 방향 수립

사용자와의 대화를 통해 방향성을 구체화한다.

다루어야 할 것:
- 접근 전략: 어떤 방식으로 풀 것인가
- 제약: 무엇을 하지 않을 것인가
- 트레이드오프: 무엇을 취하고 무엇을 포기하는가
- 경계: 이 설계의 범위는 어디까지인가

사용자가 구현 수준의 이야기를 시작하면(파일 구조, API 시그니처, 구체적 라이브러리 등)
shape이 충분히 되었다는 신호이다. Step 3으로 넘어간다.

## Step 3 — 산출물 작성

### Concept 생성/수정

새로운 도메인 엔티티가 식별되면 `.loom/concepts/`에 파일을 생성한다.
기존 concept이 변경되면 해당 파일을 수정한다.

템플릿: `references/concept-template.md`를 읽어 사용한다.

### Decision 기록

설계 과정에서 의미 있는 결정이 발생하면 `.loom/decisions/`에 기록한다.

의미 있는 결정의 기준: 대안이 존재했고, 선택에 근거가 필요했던 것.
자명한 선택은 기록하지 않는다.

템플릿: `references/decision-template.md`를 읽어 사용한다.

### 파일 이름 규칙

- concept: `{컨셉-이름}.md` (예: `user-authentication.md`)
- decision: `{NNNN}-{결정-제목}.md` (예: `0001-session-based-auth.md`)

decision은 순번을 부여하여 시간순을 보존한다.

## Summary

shape 완료 시 다음 형식으로 정리한다:

```
## Shape Summary
- 입력: [원래 요구사항 한 줄 요약]
- Concepts: [생성/수정된 concept 파일 목록]
- Decisions: [기록된 decision 파일 목록]
- 다음 단계: [plan 진행 여부와 범위]
```
