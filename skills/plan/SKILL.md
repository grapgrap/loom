# Plan

설계를 실행 가능한 태스크와 의존관계로 구조화한다.

**plan은 산출물이다. 구현하지 않는다.**

## Step 0 — 컨텍스트 로드

1. 작업 대상 프로젝트의 `.loom/index.md`를 읽어 도메인 전체 지도를 파악한다.
2. `aeira search -s {프로젝트의 .loom 경로} "{입력 키워드}"` 로 관련 문서를 식별한다.
3. 식별된 문서에서 `aeira graph neighbors -s {프로젝트의 .loom 경로} "{node}"` 로 연결 문서를 탐색한다 (outgoing: 연관 concept과 flow, incoming: 참조 decision).
4. 탐색된 concept, flow, decision을 읽는다.

shape에서 넘어온 경우, shape summary를 입력으로 사용한다.
triage에서 바로 넘어온 경우, 원본 요구사항을 입력으로 사용한다.

## Step 1 — 룰 카탈로그 로드

`.loom/rules/index.md`를 읽어 어떤 룰과 컨벤션이 있는지 카탈로그를 인지한다.

rule은 이 프로젝트에서 항상 지켜져야 하는 강제적 합의, convention은 강하게 권장되지만 대체 가능한 형태이다. 인덱스를 통해 항목 목록과 각 항목의 호출 상황을 파악한다. 본문은 활용 시점에 필요한 항목을 골라낸 뒤 로드한다.

## Step 2 — 분해

입력을 실행 가능한 태스크로 분해한다.

concept과 flow가 분해의 바탕이다 — flow의 단계는 어떤 도메인을 어떤 순서로 구현할지의 단서가 되고, concept은 각 태스크의 범위를 한정한다.

분해의 원칙:
- 각 태스크는 독립적으로 실행 가능한 단위여야 한다
- 태스크 간 복잡도가 대체로 균일해야 한다
- 한 태스크가 너무 크다고 느껴지면 더 쪼갠다

### 룰 카탈로그 활용

분해 시, 인지한 룰 카탈로그를 도구 선택 절차로 활용한다:

1. 지금 수행하려는 행동을 의식적으로 명시한다
2. 카탈로그에서 그 행동에 적용될 가능성 있는 항목을 골라낸다
3. 골라낸 항목의 본문 파일을 로드해 적용 방식을 파악한다
4. 본문에 따라 태스크를 분해한다

## Step 3 — 의존관계 식별

태스크 간의 의존관계를 명시한다.

- 어떤 태스크가 다른 태스크를 선행해야 하는가
- 병렬 진행이 가능한 태스크 그룹은 무엇인가
- 크리티컬 패스는 어디인가

의존관계는 사용자가 우선순위와 병렬 진행 가능성을 판단하기 위한 정보이다.

## Step 4 — 초안 산출

`.loom/plans/`에 plan 초안을 작성한다.

이 산출은 초안이다 — Step 5의 검증과 교정을 거치며 갱신되는 살아있는 문서다. 검증에서 분해나 의존이 바뀌면 이 파일을 직접 고친다. 확정과 그래프 갱신은 Step 5에서 하므로 이 단계에서는 `aeira sync`를 실행하지 않는다. 분해의 해상도는 작성 시점을 반영하며, 초기 plan이 거친 것은 자연스럽다.

템플릿: `../../templates/plan.md`를 읽어 사용한다.

### 파일 이름 규칙

`{plan-제목}.md` (예: `user-auth-implementation.md`)

## Step 5 — 검증

초안의 각 태스크를 사용자와 함께 검증한다. 검증의 목적은 두 가지다 — 각 태스크의 거시적 실행 가능성을 확인하는 것, 그리고 사용자가 plan으로 변경되는 범위를 정확히 인지하고 그 변경이 옳은지 판단하게 하는 것이다.

검증은 사용자와의 공동 작업이다. 에이전트 단독 점검으로 대체하지 않는다 — 사용자가 변경 범위를 인지하지 못하면 변경의 정당성을 판단할 수 없기 때문이다.

### 검증의 단위와 깊이

전 태스크를 대상으로, 큰 그림(전체 순서와 의존관계)을 유지한 채 본다. 각 태스크에 대해:

- 닿을 거점이 실재하고 접근 가능한가 — 이 태스크가 변경할 코드나 구조가 실제로 거기 있는가
- 의존이 끊기지 않는가 — 선행 태스크의 산출물이 이 태스크의 입력으로 이어지는가
- 변경이 옳은가 — 사용자에게 변경 범위를 보이고 그 타당성에 대한 판단을 구한다

깊이는 거점 확인 수준에 머문다. 코드 수준의 구체적 변경 방식은 task가 작업 직전에 합의한다 — 여기서 거기까지 내려가지 않는다. 이것이 plan의 검증(거시)과 task의 방향성 합의(미시)를 가르는 경계다.

검증에서 드러난 문제는 성격에 따라 대응이 갈린다. 태스크의 구조는 옳으나 실행할 해상도가 모자라면(입출력이나 계약이 비었으면) 표식만 남기고 실행 시점으로 위임한다. 거점이 없거나(닿을 코드·구조의 부재) 의존이 끊겼거나 변경이 부당하면, 즉 구조 자체가 틀어졌으면 교정 루프로 분해·의존을 직접 고친다.

### 해상도 부족의 표식

다음 신호가 보이는 태스크는 검증에서 깊이 파고들지 않고, 해소 경로의 표식만 남긴다:

- 태스크 기술이 "~를 정의한다", "~를 구현한다" 수준에 머물고 입출력의 형태가 명시되지 않은 경우
- 하나의 태스크에서 새로운 추상화를 도입하면서 그 추상화의 인터페이스 계약이 없는 경우
- 데이터 모델링 태스크의 결과가 후속 태스크에 반영되지 않은 경우

표식은 두 갈래다 — concept이나 흐름의 해상도가 부족하면(책임·정책·단계가 흐리면) 실행 전 shape로, 구조는 합의됐고 구현 수준의 시그니처나 형태만 미정이면 task로 보낸다. 외부 계약처럼 양쪽에 걸치면, 그 미정이 도메인 판정 자체를 바꾸면 shape로, 호출 코드 수준에 그치면 task로 가른다. plan 작성 시점에는 표식만 남기고, 실제 해소는 실행 시점에 해당 스킬이 수행한다.

### 교정 루프

검증은 마지막 통과 의례가 아니다. 거점 부재, 의존 단절, 변경의 부당함이 드러나면 분해(Step 2)나 의존관계 식별(Step 3)로 되돌아가 조정하고 재검증한다.

### 검증과 확정

검증의 산출은 교정된 plan 초안과 해소 필요 표식이다 — 사용자와의 확인 대화 자체는 문서에 적지 않는다. 검증과 교정으로 초안이 갱신되며, 사용자가 각 태스크의 방향성에 동의하면 그 갱신된 초안을 최종본으로 확정한다.

### 그래프 갱신

확정 직후, `.loom/` 내 생성·수정된 문서에 대해 `aeira sync -s {프로젝트의 .loom 경로}` 를 한 번 실행한다.

## Summary

plan 완료 시 다음 형식으로 정리한다:

```
## Plan Summary
- 범위: [이 plan이 다루는 것]
- 태스크 수: [N개 태스크, M개 phase]
- 크리티컬 패스: [가장 긴 의존 체인]
- 검증: [검증 완료, 또는 해소 필요로 표식된 태스크와 경로]
- 열린 질문: [해소되지 않은 것들]
```
